# Evaluation Approaches: Python Code vs Observe Skill

This document explains the two evaluation approaches available for the Zava Travel Concierge agent and when to use each.

---

## Overview

| Dimension | Python Code (`run_evaluation.py`) | Observe Skill (Foundry Toolkit) |
|---|---|---|
| **Dataset** | Your own static JSONL file | Auto-generated by LLM from agent config |
| **Execution** | Runs locally on your machine | Runs server-side in Foundry |
| **Results storage** | Local JSON file | Persisted in your Foundry project |
| **Portal visibility** | No | Yes (Foundry portal, across runs) |
| **Custom evaluators** | Yes (plug in any callable) | No (built-in evaluators only) |
| **Downstream integration** | Manual | Feeds prompt optimizer and continuous eval |
| **Cost** | Tokens consumed locally | Tokens consumed in Foundry |
| **Reproducibility** | Fully deterministic (custom) / near-deterministic (LLM) | Varies (dataset regenerated each run) |

---

## Approach 1 — Python Code (`run_evaluation.py`)

### How it works

You call `azure.ai.evaluation.evaluate()` locally, supplying:
- A static JSONL dataset (`data/jsonl/evaluation_data.jsonl`)
- Evaluator callables (custom and/or built-in)
- Column mappings that tell each evaluator which fields to read

```
Your machine
  │
  ├─ loads evaluation_data.jsonl
  ├─ runs CompletenessEvaluator  ──── (pure Python, no API call)
  ├─ runs DataAccuracyEvaluator  ──── (pure Python, no API call)
  ├─ runs PolicyComplianceEvaluator ─ (pure Python, no API call)
  │
  ├─ runs RelevanceEvaluator     ──── Azure OpenAI API call ──► gpt-4.1-mini
  ├─ runs GroundednessEvaluator  ──── Azure OpenAI API call ──► gpt-4.1-mini
  ├─ runs CoherenceEvaluator     ──── Azure OpenAI API call ──► gpt-4.1-mini
  └─ runs FluencyEvaluator       ──── Azure OpenAI API call ──► gpt-4.1-mini
                                           │
                                    results returned locally
                                    written to builtin_eval_results.json
```

### Custom evaluators (code-based, no LLM)

These three evaluators use deterministic logic — no model calls, no cost:

| Evaluator | Logic | Score |
|---|---|---|
| `CompletenessEvaluator` | Keyword-matches query for flight/hotel/car; checks response covers each requested service | 0–5 |
| `DataAccuracyEvaluator` | Extracts `ZV-FL-*` / `ZV-HT-*` / `ZV-CR-*` IDs from response; cross-checks prices against CSV catalog | 0–5 |
| `PolicyComplianceEvaluator` | Checks unavailable items aren't recommended without a caveat; checks no negative prices | 0–5 |

**Rule of thumb:** use code-based evaluators whenever the criterion can be expressed as a rule or a lookup. Only use an LLM evaluator when judgment is required.

### Built-in LLM evaluators

These ship with `azure-ai-evaluation` and each sends a scoring prompt to your deployed model:

| Evaluator | What the LLM judges | Prompt pattern |
|---|---|---|
| `RelevanceEvaluator` | Does the response answer the user's question? | Scores 1–5 with explanation |
| `GroundednessEvaluator` | Is the response grounded in the provided context? | Scores 1–5 with explanation |
| `CoherenceEvaluator` | Is the response logically structured and coherent? | Scores 1–5 with explanation |
| `FluencyEvaluator` | Is the response natural and well-written? | Scores 1–5 with explanation |

### Running it

```bash
# Custom evaluators only (free, no credentials beyond .env)
python evaluation/run_evaluation.py

# Custom + built-in LLM evaluators
python evaluation/run_evaluation.py --builtin

# Write results to a JSON file
python evaluation/run_evaluation.py --output evaluation/results.json

# Use a different dataset
python evaluation/run_evaluation.py --data path/to/my_data.jsonl
```

### Required environment variables (`.env`)

```
AZURE_AI_PROJECT_ENDPOINT=https://<account>.services.ai.azure.com/api/projects/<project>
AZURE_AI_MODEL_DEPLOYMENT_NAME=gpt-4.1-mini
```

### Output

- Terminal: per-row table + per-metric averages and pass rates
- `evaluation/builtin_eval_results.json`: full row-level results (when `--builtin` is used)

|---|---|---|

## Approach 2 — Observe Skill (Foundry Toolkit)

### How it works

The observe skill automates the full evaluation lifecycle from within GitHub Copilot Chat:

```
Copilot Chat (observe skill)
  │
  ├─ reads agent.yaml + system prompt
  ├─ generates diverse test dataset (LLM-driven, tailored to your agent)
  │      ↳ flight queries, hotel queries, car queries, multi-component,
  │        edge cases, out-of-scope, policy violations, etc.
  │
  ├─ sends each prompt to your deployed Foundry-hosted agent
  │      ↳ agent: zava-concierge (running in ACR/ACI)
  │
  ├─ scores each response with built-in evaluators (server-side)
  │
  ├─ analyzes failure patterns
  └─ stores results in Foundry project (visible in portal, queryable)
                    │
                    ├─ feeds Prompt Optimizer (Lab 3)
                    └─ feeds Continuous Evaluation
```

### Invoke it

In GitHub Copilot Chat:
```
Use the observe skill to evaluate my Zava Travel Concierge agent
```

### What it generates

- A diverse test dataset covering your agent's full capability surface, including edge cases you might not have written manually
- Evaluation results for each prompt across all built-in metrics
- A failure analysis summary with patterns (e.g. "fluency fails on multi-city itineraries")
- Optimization recommendations ready to feed into the prompt optimizer

### Results storage

Results are persisted **in your Foundry project** — not just locally. This means:
- You can compare runs over time in the Foundry portal
- Results are available to the prompt optimizer skill
- You can set up continuous evaluation to run on a schedule

## When to use each

| Scenario | Recommended approach |
|---|---|
| CI/CD quality gate on a known golden dataset | Python code |
| Domain-specific checks (prices, IDs, policy) | Python code + custom evaluators |
| Discovering unknown failure modes | Observe skill |
| Exploring edge cases you haven't thought of | Observe skill |
| Tracking quality trends over time in the portal | Observe skill |
| Running without Foundry portal access | Python code |
| Feeding results into the prompt optimizer | Observe skill |
| Offline / air-gapped environment | Python code (custom evaluators only) |

---

## How they complement each other

The recommended workflow is to use **both**:

1. **Observe skill** — run first to discover failure patterns and generate a rich dataset
2. **Export or recreate** that dataset as a JSONL file
3. **Python code** — add your custom evaluators on top of the generated dataset for domain-specific checks the observe skill cannot do
4. **Iterate** — after prompt optimization (Lab 3), re-run both to measure improvement

```
Observe skill ──► identifies failures ──► prompt optimizer ──► improved agent
                                                                      │
Python code ──── verifies domain rules ◄──────────────────── re-evaluate
(custom evals)   still hold after optimization
```
